  First, the good news

  This is a rework, not greenfield. A lot already exists: per-job SignalR
  groups, a JobRunner with per-repo write-locking, a JobSink that already
  aggregates byte counters, event forwarders for every Core event the design
  names, the cost-approval handshake (RestoreApprovalRegistry), a cron
  SchedulerService, and a SQLite job store. Events already carry byte sizes. So
  the design's "no Core changes" constraint holds вҖ” everything is derivable from
  what Core emits today.

  But the design is ~95% a visual/UX spec. The genuinely unresolved stuff is
  almost entirely on the Api state-model and job-lifecycle side, plus a few
  behavior contradictions with how things work today. Here's what I found,
  grouped.

  A. The big architectural forks (these gate everything else)

  1. Where does live job state live, and what survives an Api restart?
  Today JobSink holds the byte aggregates in memory, per running job, and there
  is no way to fetch a snapshot вҖ” a new/returning connection gets nothing. The
  design's whole "reattach / pill reconnects on revisit / send a full snapshot
  on subscribe, then deltas" hinges on state a second connection can read. And
  running jobs are fire-and-forget Tasks: on Api restart the job dies but the DB
  still says running. This is the decision everything else depends on.

  2. Reattach protocol + per-job message routing.
  Bigger than it looks: RealtimeService has a single global connection with 
  global log$/progress$/cost$/done$ subjects вҖ” messages aren't tagged with a
  jobId. The client silently assumes one active job (the drawer's). For the pill
  + multiple jobs + a /jobs/:id page that can attach to any job, every SignalR
  message needs a jobId, and the hub needs a new AttachToJob(jobId) (join group
  + return current aggregate). Open: snapshot via REST GET /jobs/{id}
  first-then-deltas, vs a SignalR Snapshot message on attach.

  3. Rehydration auto-resume вҖ” the second-biggest gap.
  The current RestoreCommandHandler does not wait hours for rehydration. It
  downloads what's available now, requests rehydration for the rest, and returns
  with ChunksPendingRehydration > 0 (idempotent per ADR-0017 вҖ” you re-run to
  pick up hydrated chunks). So the design's "Automatically restore as chunks
  become available" toggle + adaptive-polling BackgroundService requires new Api
  orchestration that re-drives RestoreCommand when chunks flip available. Open:
  re-invoke the whole command (re-walks the tree)? persist restore params
  (repo/version/targetPaths/priority/destination/toggle вҖ” none are in JobRecord
  today)? and on auto-resume do we re-prompt for cost (no вҖ” already paid; pass a
  non-prompting confirm)?

  4. Cost-confirmation lifecycle is a behavior change, not just a redraw.
  Today, closing the tab declines/cancels a pending cost approval
  (OnDisconnectedAsync вҶ’ CancelForConnection, and DrawerStore.close() sends
  decline). The new design says a blocked job parks in "Needs your attention" 
  and waits вҖ” decoupled from the connection. Those contradict. We need to
  decide: does navigating away / closing the tab still cancel, or does the job
  now wait indefinitely?

  B. Smaller behavioral clarifications

  - 5. Cancellation вҖ” "Cancel job" is in the design but no cancellation token is
  plumbed (JobRunner passes CancellationToken.None). Need per-job CTS +

  - 6. Progress firehose вҶ’ coalesced deltas + ETA window вҖ” forwarders currently

вҸә #3 вҖ” Rehydration auto-resume orchestration

  This is the other heavy one, and there's a latent bug in today's code that
  frames it. Right now RunRestoreAsync unconditionally does CompleteJob(jobId, 
  "completed", вҖҰ, "Restore complete.") after the command returns вҖ” it never 
  checks result.ChunksPendingRehydration. So a restore that requested
  rehydration for 282 archive-tier chunks is marked completed immediately, and
  those chunks are silently dropped; the user has to notice and manually re-run
  hours later. The design's whole rehydration story is really "make this correct
  + automatic."

  The grain of the mechanism: re-drive the whole command

  The RestoreCommandHandler is already idempotent (ADR-0017): re-running it
  re-resolves the snapshot, re-classifies (now more chunks are hot), downloads
  the newly-available ones, skips already-restored files (conflict routing вҶ’
  SkipIdentical), and re-requests still-pending chunks (stage 5 skips chunks
  already pending). So auto-resume = re-invoke RestoreCommand on a schedule 
  until it returns ChunksPendingRehydration == 0. Zero Core changes; the
  re-drive is the authoritative status check.

  I considered a finer mechanism вҖ” persist the resolved chunk set and download
  only the pending subset вҖ” but that means duplicating resolution logic in the
  Api or changing Core. Reject it. Re-drive the whole command.

  What's already in place

  - The codebase already has the "cancel вү  failure" idiom:

  - cancelled is already a documented job status; CompleteJob takes a free-form

  So #5 is mostly plumbing a token that already has a socket to plug into.

  Archive cancel is provably safe

  The snapshot is stage 6d вҖ” the very last step, created only after all uploads
  and tree-build (and skipped if unchanged). So cancelling mid-upload throws OCE
  before any snapshot is published вҶ’ the repository state is unchanged and 
  consistent. The chunks uploaded so far become orphans that a future archive 
  reuses via dedup вҖ” not corruption, not wasted work (this is the same
  partial-state the durability model already tolerates for a crash, per
  AGENTS.md "blob storage is non-transactionalвҖҰ recoverable"). Restore is
  likewise safe: files already written stay on disk, re-running continues
  (idempotent).

  The design

  Per-job CancellationTokenSource registry (in-memory, keyed by jobId, alongside
  the JobState registry from #1). JobRunner creates the CTS at run start,
  passes cts.Token into CreateJobProviderAsync + mediator.Send, and
  disposes/deregisters in finally. In-memory only вҖ” a token can't be persisted,
  and after a restart there are no live runs to cancel.

  Hub CancelJob(jobId) handles both live and parked jobs:
  - Live run (CTS present) вҶ’ cts.Cancel() вҶ’ OCE propagates вҶ’ the finally still
  releases the repo gate вҶ’ JobRunner maps OCE вҶ’ cancelled.
  - Parked (no CTS): awaiting-cost вҶ’ same as decline; waiting-rehydration вҶ’
  disarm the poller (#3) and mark cancelled.

  Distinguish cancelled from failed вҖ” the single behavioral fix in JobRunner:
  its current catch (Exception) maps everything (including OCE) to failed.
  Change to mirror the existing idiom вҖ” catch (OperationCanceledException) вҶ’ 
  CompleteJob(вҖҰ, "cancelled", вҖҰ), other exceptions вҶ’ failed.

  #6 вҖ” Progress throttling + ETA

  Three distinct pieces here. The first is the one that fixes the original
  complaint (the bar that "jumps to 95% and hangs").

  6a. Shift the aggregate model from counts вҶ’ bytes

  Today JobSink counts files (_filesHashed, hashed*90/total) вҖ” which is exactly
  why the bar lurches: hashing 95% of the files finishes fast, then the long
  upload has no headroom left. The design's layered bar is byte-weighted:
  scannedBytes (from FileScannedEvent) вҠҮ hashedBytes вҠҮ uploadedBytes, with
  totalBytes from ScanCompleteEvent. The headline % becomes uploadedBytes / 
  totalNewBytes вҖ” monotonic and honest.

  One concrete wrinkle to verify early (it bumps against "no Core changes"): the
  three layers must be in the same unit вҖ” original dataset bytes вҖ” or the
  uploaded layer, measured in stored (compressed) bytes, would top out at ~half
  the track and look permanently stalled. Reconciling:
  - Tar chunks: easy вҖ” TarBundleSealingEvent carries UncompressedSize + TarHash;
  on TarBundleUploadedEvent(TarHash) add the remembered uncompressed size.
  - Large chunks: ChunkUploadedEvent gives only StoredSize. We need each
  uploaded large chunk's original size. Action: confirm that's derivable from 
  existing events (ChunkUploadingEvent.Size, or by remembering the file size
  from FileHashing/FileScanned). If it genuinely isn't, that's the one spot that
  would otherwise want a Core event tweak вҖ” so we want to know now, not
  mid-build.

  Also tracked here: dedupedBytes/files (purple KPI) and per-stage counters.

  6b. Coalesce the firehose вҶ’ ~1 s absolute-state deltas

  Right now every FileHashedEvent/ChunkUploadedEvent/etc. calls
  SendAsync("Progress") вҖ” thousands of sends per run. New model: forwarders only
  mutate the in-memory JobState; they never send. A per-job ~1 s timer (living
  in the JobState registry from #1) emits one Progress message carrying the
  current absolute aggregate, plus an immediate emit on transitions (stage
  complete, cost prompt, done) for snappiness. Because deltas are absolute-state
  (#2), coalescing is trivially safe вҖ” you just send the latest. This same
  timer is the natural home for #1's throttled StateJson flush (a slower ~5вҖ“10 s
  sibling) and for 6c's sampling below. All Api-side; no Shared calls.

  6c. ETA + throughput via a sliding window

  - Each ~1 s tick pushes (now, uploadedBytes) into a small ring buffer. Rate =
  О”bytes / О”time over the last ~60 s window (a sliding window, not a cumulative
  average вҖ” the latter lags badly when throughput changes). That rate is the
  "2.4 MB/s sustained" throughput readout.
  - ETA = (totalNewBytes вҲ’ uploadedBytes) / rate. But totalNewBytes is only
  known once hash+dedup-route completes (~90 s in) вҖ” so show "estimatingвҖҰ" until
  then, exactly as the design specifies. The throttle timer doubles as the ETA
  sampler, so this is nearly free.
  - Rehydration "resumes ~03:40" is the other ETA and stays a heuristic off the
  consolidated wait-window const (from the #4 consolidation) вҖ” not rate-based.

  The resulting Progress payload

  Grows from {pct, stats} to the full absolute aggregate вҖ” { jobId, phase, 
  scannedBytes, totalBytes, hashedBytes, uploadedBytes, dedupedBytes, 
  etaSeconds|null, throughputBytesPerSec, perStage[], warningCount } вҖ” and the 
  snapshot-on-attach (#2) returns this exact shape, so the client has one render
  path whether it's live-delta or reattach.

  ---
  Net effect: byte-weighted honest progress, ~1 msg/s instead of thousands, a
  windowed ETA that says "estimatingвҖҰ" until it can be truthful. The only thing
  needing verification is the large-chunk original-size question in 6a вҖ” worth
  checking against the events before we commit the "no Core changes" promise.

вҸә That's a good call, and it's stronger than the current behavior вҖ” but there's
  a subtlety that makes it more than just the existing gate, and getting it
  right is what actually collapses #8.

  The guard must cover parked jobs, not just running ones

  Today JobRunner has a per-repo SemaphoreSlim that serializes the active 
  execution вҖ” and it queues a second job rather than rejecting it. Two problems
  for what you want:
  1. You want reject, not queue ("a scheduled archive must fail if a one-off is
  still running").
  2. Parked jobs release the gate (#3/#4): a restore waiting hours for
  rehydration, or sitting in awaiting-cost, is not holding the semaphore. So a
  plain execution-gate would happily let a scheduled archive start alongside a
  parked restore вҖ” exactly the "restore coinciding with archive" you're trying
  to forbid.

  So the guard can't be the in-process semaphore. It has to be a
  repository-level invariant over job state in the DB:

  в–Һ A repository may have at most one job in a non-terminal state (running | 
  в–Һ awaiting-cost | waiting-rehydration). Any new start request вҖ” one-off, 
  в–Һ scheduled, or restore вҖ” is rejected while such a job exists.

  Terminal states (completed/failed/cancelled) don't block. And the
  resume/re-trigger of the existing job is exempt вҖ” it's the same jobId
  transitioning states (#3/#4), not a new operation.

  Enforcement
