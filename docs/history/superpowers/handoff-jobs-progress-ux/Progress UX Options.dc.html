Turn 3 · layered bar, log-calibrated

One bar, phases as deepening shades

Start → drawer dismisses → 1c pill → "View job" opens these pages. One track, layered fills: each phase runs ahead of the next in a lighter shade, all measured in bytes of the same dataset — so the bar never jumps or hangs. Bundling (tar-build/tar-upload) indeed doesn't fit as its own layer: it's transport, not progress against the source — its bytes count toward Upload the moment each tar seals and uploads. Numbers and timings below come from the 2026-06-28 log (3,122 files · 3.16 GB · scan 96 s · upload 22 min · finalize 32 s).

Turn 1 · 3 directions

Archive / Restore progress — replacing the terminal

All three are Api/Web-only: they aggregate the events Arius.Core already emits (FileScanned/ScanComplete, FileHashed, TarEntryAdded/TarBundleSealing, ChunkUploading/Uploaded, RehydrationStatus, ChunkDownloadCompleted, FileRestored…) into byte-weighted progress, ETA and MB/s. The console is gone in every option. Each option shows the drawer (or files page) and the Jobs detail page built from the same component.

1a

Pipeline stages

Progress = five stages (Scan → Hash → Bundle → Upload → Snapshot), each with its own counters and bar. The overall bar is byte-weighted by upload volume, so no more 95%-hang: the hang was Scan/Hash finishing instantly while Upload crawled — here Upload owns its own bar. Drawer and job page are the same component at two widths.
