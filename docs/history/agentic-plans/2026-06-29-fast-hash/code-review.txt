  ūüĒī Critical ‚ÄĒ silent data loss under --fast-hash

  These are the cardinal sin for a backup tool: a file is changed but archived
  under its old hash, or dropped entirely. The spec calls
  "unchanged-when-actually-changed" the only unsafe misprediction ‚ÄĒ two of these
  are exactly that.

  1. ctime captured after the content read ‚ÄĒ
  src/Arius.Core/Features/ArchiveCommand/ArchiveCommandHandler.cs:402
  FullHashAndRecordAsync reads+hashes the file (line 400), then reads
  ctime/inode/dev (line 402) and Record()s them together. A same-machine write
  landing between line 400 and 402 (an appended log, a same-size in-place DB
  write) stores the post-modification ctime with the pre-modification hash. Next
  --fast-hash run: ctime fast-lane matches ‚Üí declares UNCHANGED ‚Üí reuses the
  stale hash ‚Üí the changed content is never re-uploaded and is permanently
  absent from later snapshots. Fix: capture signals before the read, or re-check

‚Źļ I've now read all the relevant code, tests, and docs. Before I start changing
  things, here are the answers to your analysis questions ‚ÄĒ and two decisions I
  genuinely need from you.

  Analysis answers

  (1) ctime captured after the read ‚ÄĒ why does flipping the order fix it, and 
  what's the real exposure?

  True atomicity (stat + read as one operation) isn't achievable and isn't
  needed. What matters is which direction the residual race resolves:

  - Today (signals after the read): a write that lands during the read gets
  folded into the ctime we then store. Next run, the live ctime equals the
  stored ctime ‚Üí HIT ‚Üí the torn/stale hash is reused permanently. Unsafe.
  - Flipped (signals before OpenRead): the stored ctime becomes a conservative
  lower bound on "the time the bytes I hashed were valid." Any write at or after
  that snapshot necessarily advances ctime past the stored value ‚Üí next run
  MISS ‚Üí re-hash. Safe.

  So the real exposure today is not the ~¬Ķs between lines 400‚Üí402 ‚ÄĒ it's the
  entire read window (OpenRead 396 ‚Üí signal capture 402). Every byte is read
  before we snapshot ctime, so a write anywhere during the read (seconds, for a
  large file) is baked into the stored ctime and masks the stale hash forever.
  After the flip the residual is essentially nil ‚ÄĒ the only unfixable case is
  two changes within the same ctime tick (nanoseconds), which is inherent to any
  ctime/mtime scheme. Fix: capture signals (and mtime, see #12) immediately 
  before OpenRead.

  (4) Shrinking-file fingerprint ‚ÄĒ real exposure?

  The window is GetFileSize (line 326, top of the per-file Stage-2 worker) ‚Üí
  read completion (line 400) for that one file ‚ÄĒ bounded per file, though
  seconds for a large file being actively truncated. But unlike #1, this one is 
  self-healing: the row stores Size = the stale (larger) value while the
  uploaded bytes/hash are the actual (shorter) content ‚ÄĒ and the chunk is
  self-consistent (hash matches the bytes actually read). Next run, live size ‚Č†
  stored size ‚Üí MISS ‚Üí re-hash. So it's a transient cache poisoning, not data
  loss. I'd downgrade #4 from my original ranking. Optional cheap hardening
  (re-stat / use the opened stream's length so size‚ÜĒfp‚ÜĒhash are mutually
  consistent and the entry isn't wasted) ‚ÄĒ I'll leave it unless you want it.

  (6) Is a friendly "delete the folder" exception consistent with ChunkIndex?

  No ‚ÄĒ and that divergence is correct. ChunkIndex silently recreates the DB
  (RecreateDatabase) because it has a remote-backed repair path. The hashcache
  has neither remote backing nor a repair command, and you're not comfortable
  with silent repair yet ‚ÄĒ so it should hard-fail with a friendly message 
  telling the user to delete the hashcache folder (one full-hash run recovers).
  What stays consistent with ChunkIndex is the exception shape (a dedicated
  ‚Ä¶LocalStoreException wrapping the SqliteException with actionable guidance);
  what diverges is recovery (throw-and-instruct vs silent recreate). One extra
  thing I'll handle: Stage 2's per-file catch currently swallows everything and
  skips the file, so a corrupt-DB throw mid-run would silently drop files ‚ÄĒ I'll
  make this exception bypass that catch so corruption faults the run loudly
  instead.

  (12) Record the file's mtime, not run wall-clock ‚ÄĒ right?

  Yes. I'll store the file's real last-write time (already available via
  fs.GetTimestamps), captured before the read alongside #1's signals, so the
  documented mtime column and the future mtime-prefilter seam are actually
  usable.

  (14) Do the two fingerprint paths each have a purpose?

  Yes, genuinely ‚ÄĒ they're opposite I/O patterns, neither subsumes the other:
  - ComputeBySeeking ‚ÄĒ cold floor path: opens and seeks to only the sample
  regions (~k√ó256 KiB), hoping to avoid a full read.
  - Sampler (via SparseSamplingStream) ‚ÄĒ warm path: captures those same regions
  for free as bytes stream past during a full re-hash.

  They must produce byte-identical digests (size ‚ÄĖ region-bytes). I'll add
  cross-referencing "keep in sync" comments on both and point at the existing
  Sampler_MatchesSeekingFingerprint_ForSameContent test that guards it.

  (8) Synology ‚ÄĒ are we always re-hashing the whole file?

  No. If the stat fallback doesn't bind, TryGet returns null ‚Üí you lose only the
  zero-read ctime fast-lane. The sparse-fingerprint floor still works
  (ComputeBySeeking reads just the sample regions and reuses the cached hash on

  files get fully re-hashed. The catch: files in (256 KiB, 1 MiB] hit the #3
  buffer bug on the floor ‚Üí currently throw ‚Üí file dropped. So fixing #3 is

  binding is that older glibc (Synology DSM) exports the stat syscall only
  through the versioned __xstat symbol, not a bare stat ‚ÄĒ which is exactly
  decision #2 below.
