  The key finding that dissolves the dilemma

  The failure is only a one-time write problem. Azure forbids Set Blob Metadata
  on Archive-tier blobs (your 409 BlobArchived), but:

  - Reading metadata on archived blobs is allowed ‚ÄĒ so repair could already read
  archived blobs if the descriptor were on them.
  - Azure explicitly allows Set Blob Tags on archived blobs (confirmed on MS
  Learn). Blob index tags are a per-blob, co-located, durable k/v store ‚ÄĒ the
  same robustness property that makes metadata your repair source-of-truth
  today. They are not "the chunk index": losing the index doesn't lose them,
  because they live on each blob.

  Your two descriptor facts that matter:
  - The hot path (restore/list/download) reads the chunk index, never blob
  metadata ‚ÄĒ so it's unaffected.
  - Only repair (ChunkIndexService.RepairAsync) reads per-blob descriptors, by
  listing chunks and reading item.Metadata. That's the single place to teach
  "read metadata, fall back to tags."

  All your descriptor values (arius_type = large/tar/thin, decimal sizes,
  lowercase-hex parent hash) fit the tag charset and the 10-tag limit. Only
  large and tar chunks can be archived; thin stubs are created fresh at Cool
  tier, so they're untouched.

  So the tags approach satisfies both your constraints: no rehydration, and
  repair stays per-blob durable (not index-dependent).

  The one honest caveat: Azurite can't reproduce real Archive-tier behavior (the
  test suite fakes it), so the "write succeeds on an archived blob" step is
  verified against real Azure (re-running the ariusciuse migration); the repair
  metadata-or-tags fallback gets a deterministic unit test.

  The cost number, and why it matters for this tool specifically

  Tags carry a recurring charge (‚Čą $0.10 per 10,000 tags/month, region-dependent
  ‚ÄĒ confirm on the pricing page; the research had a 10√ó arithmetic slip, the
  corrected figures are below). Metadata is free.

  At ~3 tags/chunk: ~$3/mo per 100k chunks, ~$30/mo per 1M chunks, ~$300/mo per 
  10M ‚ÄĒ forever.

  Archive storage itself is ~$1/TB/month. So for a cheap-archival tool,
  all-chunk tags can plausibly cost more than the archive storage they describe.
  That's the catch.

  The reframing that I think changes the decision

  Native v7 never actually needs tags. Your upload path sets metadata while the 
  blob is still Hot/Cool, then tiers to Archive
  (ChunkStorageService.cs:133-134). Metadata persists through archival
  (read-only, but readable), so repair reads it fine even on archived chunks.
  You only physically can't write metadata when the blob is already archived 
  before you write the descriptor ‚ÄĒ which is exactly and only the migration case
  (a v5 repo that's been sitting in Archive).

  So tags solve a migration-only problem. Going all-tags pays a perpetual,
  per-chunk cost to simplify code for a situation that, in native v7, doesn't
  arise.

  My recommendation

  Keep metadata as the descriptor (free, unchanged write path, already
  battle-tested), and add tags purely as the archive-writable carrier the
  migration uses for already-archived blobs ‚ÄĒ with repair reading "metadata,
  else tags." Native v7 stays $0; only the subset of pre-archived migrated
  chunks ever gets tagged.

  The all-tags route is genuinely simpler code (single path, no tier branch) and
  you can make the breaking change ‚ÄĒ but you'd pay for that simplicity monthly
  on every chunk, on a tool whose whole point is cheap archival.

  that), so it'd need an artificial name like chunks/{hash}.d or
  chunks/.descriptors/{hash}.

  - ‚úÖ Repair's existing single chunks/ listing sweeps them up ‚ÄĒ no second list
  call.
  - ‚ĚĆ Pollutes a namespace several paths iterate assuming "everything here is a
  chunk." Repair's ContentHash.Parse(item.Name) + arius_type switch
  (ChunkIndexService.cs:723-758), the archived-chunk enumeration for
  rehydration/tier-stats (ChunkStorageService.cs:~236), and dedup/count logic
  would each need a new "skip the sidecar" filter ‚ÄĒ new bug surface in working
  code.
  - ‚ĚĆ The thin-stub analogy partly breaks here. Thin stubs get a naturally 
  distinct key (the file's content hash ‚Č† the parent tar's hash), so they never
  collide. A descriptor for a large/tar chunk describes the same hash, forcing a
  naming hack. And a sidecar carrying arius_type=large looks just like a real
  large chunk to the switch ‚ÄĒ repair would have to disambiguate by path, not
  metadata.

  B. Separate sibling prefix (chunk-descriptors/{hash} ‚ÜĒ chunks/{hash}) ‚ÄĒ my
  recommendation.

  - ‚úÖ chunks/ listing stays untouched ‚ÄĒ every existing chunk consumer keeps
  seeing only real chunks + thin stubs. Correct-by-construction; no new filters,
  no risk to working paths.
  - ‚úÖ Unambiguous by location ‚ÄĒ chunk-descriptors/{hash} is by definition the
  descriptor for chunks/{hash}. Clean 1:1 on the identical hash, no naming
  gymnastics, no arius_type confusion.
  - ‚úÖ Trivially inspectable and wholesale-deletable (e.g. if you ever
  rehydrate-and-rewrite real metadata, just drop the prefix).
  - ‚ĚĆ Repair adds a second list pass + a dict join by hash ‚ÄĒ cheap (repair is
  rare), and it's the same shape as the existing thin‚Üítar enrich pass
  (ChunkIndexService.cs:687-690).

  On the name: chunks-v5/ is origin-based ‚ÄĒ it brands the prefix as migration
  scaffolding and discourages native v7 from leaning on it, but bakes "v5" into
  the durable layout (a misnomer if the mechanism is ever reused).
  chunk-descriptors/ is function-based ‚ÄĒ self-documenting ("repair reads
  descriptors from here") and future-proof. I lean chunk-descriptors/, but it's
  a one-line change if you prefer to mark it as v5 scaffolding.

     Plan: Archive-safe chunk descriptors via per-chunk sidecar blobs

     Context

     The v5‚Üív7 migration (src/Arius.Migration/MigrateV5.cs) fails with
     Azure.RequestFailedException: 409 BlobArchived in Stage 3 when it calls
     SetMetadataAsync on chunk blobs that are already in the Azure Archive
     access tier (e.g. ariusciuse/v5migrationtest). Azure forbids Set Blob 
     Metadata on archived blobs.

     That metadata (arius_type, original_size, chunk_size, parent_chunk_hash ‚ÄĒ
     BlobMetadataKeys in src/Arius.Core/Shared/Storage/BlobConstants.cs) is the
     durable, per-blob source of truth that ChunkIndexService.RepairAsync
     re-reads to rebuild the chunk index. So we cannot just put it in the index
     (that breaks repair), and we will not rehydrate (cost).

     Why this approach. Native v7 never has this problem: it writes the
     descriptor while the blob is still Hot/Cool, then tiers to Archive ‚ÄĒ
     metadata persists (read-only) and repair reads it fine. The problem exists
     only for the migration of blobs that are already archived. So we need an
     archive-writable carrier used only by the migration.

     We evaluated three carriers:
     - Blob index tags ‚ÄĒ archive-writable, but carry a small perpetual per-tag
     charge, are weakly supported in Azurite, and need a new SetTagsAsync +
     tags-listing abstraction. Rejected.
     - All-metadata‚Üítags ‚ÄĒ pays the recurring cost on every chunk to solve a
     migration-only problem. Rejected.
     - Per-chunk descriptor sidecar blob (chosen) ‚ÄĒ a zero-byte Cool blob
     carrying the descriptor in its metadata, modeled on the existing thin-stub
     pattern (ChunkStorageService.UploadThinAsync). Zero-byte Cool blobs are
     billed at 0 bytes (no minimum object size), so it's ~$0 recurring, fully 
     Azurite-testable (normal blobs + metadata), and reuses a pattern already in
     the codebase.

     Layout decision. Sidecars live under a separate sibling prefix 
     chunk-descriptors/{hash} (1:1 with chunks/{hash}), not colocated under
     chunks/. This keeps the chunks/ namespace untouched ‚ÄĒ every existing
     chunk-iterating path (ChunkIndexService repair switch, ChunkStorageService
     archived-chunk enumeration, counts/dedup) stays correct-by-construction
     with no new filters. Repair pays a cheap second list pass + hash join (same
     shape as the existing thin‚Üítar enrich pass).

     Scope reminder. The native v7 upload path (ChunkStorageService) is
     unchanged ‚ÄĒ it keeps writing free metadata. Sidecars are created only by
     the migration, only for chunks found in the Archive tier. Thin stubs are
     created fresh at Cool tier and are never archived, so they keep their own
     metadata and need no sidecar. Only large and tar chunks can be archived.

     Plan: Archive-safe chunk descriptors via per-chunk sidecar blobs

     Context

     The v5‚Üív7 migration (src/Arius.Migration/MigrateV5.cs) fails with
     Azure.RequestFailedException: 409 BlobArchived in Stage 3 when it calls
     SetMetadataAsync on chunk blobs that are already in the Azure Archive
     access tier (e.g. ariusciuse/v5migrationtest). Azure forbids Set Blob 
     Metadata on archived blobs.

     That metadata (arius_type, original_size, chunk_size, parent_chunk_hash ‚ÄĒ
     BlobMetadataKeys in src/Arius.Core/Shared/Storage/BlobConstants.cs) is the
     durable, per-blob source of truth that ChunkIndexService.RepairAsync
     re-reads to rebuild the chunk index. So we cannot just put it in the index
     (that breaks repair), and we will not rehydrate (cost).

     Why this approach. Native v7 never has this problem: it writes the
     descriptor while the blob is still Hot/Cool, then tiers to Archive ‚ÄĒ
     metadata persists (read-only) and repair reads it fine. The problem exists
     only for the migration of blobs that are already archived. So we need an
     archive-writable carrier used only by the migration.

     We evaluated three carriers:
     - Blob index tags ‚ÄĒ archive-writable, but carry a small perpetual per-tag
     charge, are weakly supported in Azurite, and need a new SetTagsAsync +
     tags-listing abstraction. Rejected.
     - All-metadata‚Üítags ‚ÄĒ pays the recurring cost on every chunk to solve a
     migration-only problem. Rejected.
     - Per-chunk descriptor sidecar blob (chosen) ‚ÄĒ a zero-byte Cool blob
     carrying the descriptor in its metadata, modeled on the existing thin-stub
     pattern (ChunkStorageService.UploadThinAsync). Zero-byte Cool blobs are
     billed at 0 bytes (no minimum object size), so it's ~$0 recurring, fully 
     Azurite-testable (normal blobs + metadata), and reuses a pattern already in
     the codebase.

     Layout decision. Sidecars live under a separate sibling prefix 
     chunk-descriptors/{hash} (1:1 with chunks/{hash}), not colocated under
     chunks/. This keeps the chunks/ namespace untouched ‚ÄĒ every existing
     chunk-iterating path (ChunkIndexService repair switch, ChunkStorageService
     archived-chunk enumeration, counts/dedup) stays correct-by-construction
     with no new filters. Repair pays a cheap second list pass + hash join (same
     shape as the existing thin‚Üítar enrich pass).

     Scope reminder. The native v7 upload path (ChunkStorageService) is
     unchanged ‚ÄĒ it keeps writing free metadata. Sidecars are created only by
     the migration, only for chunks found in the Archive tier. Thin stubs are
     created fresh at Cool tier and are never archived, so they keep their own
     metadata and need no sidecar. Only large and tar chunks can be archived.

‚ĚĮ why did you do the overwrite parameter?                                       

‚Źļ Direct reason: I changed the snapshot's timestamp from DateTimeOffset.UtcNow
  to a deterministic value (the v5 state-DB name). That removed the free
  idempotency we'd been relying on.

  Before the change, every migration run stamped the snapshot with now ‚Üí a
  unique blob name each time ‚Üí CreateAsync's overwrite: false never collided.
  Now that the name is fixed (2026-06-20T180134.131Z), a second run of the 
  migration would hit the same blob name, and CreateAsync uploads with
  overwrite: false, which throws BlobAlreadyExistsException at Stage 6. We
  re-ran the migration repeatedly during this session, so that would have broken
  immediately.

  So overwrite: true makes re-running the migration on the same state
  idempotently rewrite the (identical-content) snapshot instead of failing ‚ÄĒ
  consistent with how the rest of Stage 3/4 already behaves (sidecars use
  overwrite:true, thin stubs handle existing, repair rebuilds).

  That said ‚ÄĒ it's a fair thing to question, and there are two things worth your
  call:

  1. Whether to widen the shared interface for a migration-only need. I added
  overwrite to ISnapshotService.CreateAsync (default false, so the normal
  append-only archive path is untouched). The alternative is to keep the
  interface narrow and handle the collision inside the migration ‚ÄĒ e.g. catch
  BlobAlreadyExistsException and proceed (we already know the timestamp for
  PromoteToSnapshotVersionAsync), or pre-delete the snapshot blob. That keeps
  ISnapshotService clean at the cost of a few lines of catch-logic in MigrateV5.
  2. Whether re-runs should be idempotent at all. A defensible alternative is to
  leave overwrite: false and let a second migration fail loudly ("snapshot
  already exists") ‚ÄĒ treating migration as strictly one-shot. I went with
  idempotent because re-running yields an identical snapshot (same rootHash), so
  overwriting is harmless, and a cryptic 409 mid-migration is poor UX.

  If you'd rather not touch the shared interface, I'd revert the CreateAsync
  change and catch the conflict in the migration instead. Want me to switch to
  that?
